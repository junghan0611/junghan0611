## 1. SKS Hub — 펌웨어에서 제품까지

**고퀄 · 2025.12 ~ 현재 · 양산 출하됨**

### 문제

Zigbee/Wi-Fi 게이트웨이 펌웨어를 만들라는 계약이었다.
그런데 펌웨어만 나가면 제품이 되지 않는다. 서버가 없고 앱이 없다.
계약서에 없던 두 층이 비어 있었다.

### 무엇을 골랐나

**펌웨어 — C가 아니라 Zig.**
레거시 C Zigbee SDK를 Zig의 타입 안전 FFI 층 뒤로 밀어 넣었다.
상태를 `HubState` 하나로 모으고, 전이를 순수 함수 `transition()`으로 갈랐다.
결정론적 상태 기계여야 에이전트든 사람이든 이 코드를 신뢰하고 고칠 수 있다.

**서버 — 직접 Go로 짰다.** 계약에 없었지만 없으면 제품이 아니었다.

**앱 — 직접 Flutter로 짰다.** 같은 이유다.

**프로토콜 하나, 백엔드 둘.**
연결형 배포에는 AWS IoT, 폐쇄망에는 로컬 백엔드.
**펌웨어는 양쪽에서 수정 없이 그대로 나간다.**
붙이는 설계가 아니라 갈아 끼울 수 있게 자르는 설계다.

- 상호 TLS 로컬 브로커
- 섀도 미러링 + 단조 증가 버저닝
- 루프백 이벤트 스트리밍
- 멀티 허브 팬아웃 — 실제 하드웨어에서 검증 (서버 1대, 살아 있는 허브 2대, 종단간)

### 무엇이 남았나

- **양산 출하.** 제품이 되어 나갔다.
- **30일 198 커밋** — 디바이스 드라이버 · 프로토콜 · 클라우드 연동 층을 가로질러.
  Claude를 짝 프로그래머로 썼다. [README §2.2](README.md#22-실측--에이전트와-함께-만든-결과물)
- **에이징 테스트 자동화** — 24시간 안정성 검증

### 차세대 허브 — 지금 만드는 것

출하한 허브가 감수해야 했던 제약을 넘는 작업.

| 축 | 출하 버전 | 차세대 |
|---|---|---|
| 타깃 | ARMv7 / glibc, 벤더 sysroot | **RISC-V (SG2000), 정적 링크 musl** |
| 보드 의존 | 코드가 보드를 안다 | **보드 HAL 추출** — 순수 허브 코어는 자기가 무슨 보드에서 도는지 모른다. 출하한 허브가 여러 백엔드 중 하나가 된다 |
| Zigbee 전송 | 라디오를 직접 구동 | **Zigbee2MQTT (EFR32MG24)로 위임** |
| 디바이스 모델 | 하드코딩된 7개 타입 핸들러 (64MB RAM 보드의 흔적) | **디바이스 애그노스틱** — 디바이스 모델이 일반 엔티티로 통과 |

핵심은 두 번째 줄이다. **보드 HAL을 뽑아내면 어제의 제품이 오늘의 백엔드 하나가 된다.**
이게 내가 리팩터링을 이해하는 방식이다. 지우는 게 아니라 강등시키는 것.

## 2. HomeAgent — 하네스가 터미널을 떠날 때

**고퀄 · 2025.09 ~ 현재 · 오픈소스**
→ [homeagent-config](https://github.com/junghan0611/homeagent-config)

### 문제

클라우드 없이 도는 Matter 스마트홈 허브. 그 위에 **온디바이스 AI 에이전트**.
그런데 진짜 질문은 모델 벤치마크가 아니었다.
**터미널에서 쓰던 하네스의 관심사가 엣지에서도 살아남는가** — 결정론적 제어,
사람의 오버라이드, 플랫폼 연속성, 제약된 기기에서의 로컬 우선 AI.

### 무엇을 골랐나

**하나의 Go 코드베이스에서 두 개의 완전히 다른 타깃.**

- RPi5 + Hailo-8 NPU (Yocto Linux)
- RK3576 (Android)
- **리눅스/안드로이드 배포 간 코드 공유 96%**

**단일 Go 바이너리**가 Matter 디바이스 제어 · 실시간 SSE 스트리밍 · LLM 에이전트를 전부 든다.
배포 단위가 하나여야 엣지에서 관리가 된다.

**BLE 커미셔닝**은 matterjs 브리지를 통해. 5개 디바이스 타입, OTBR 통합.

**UI는 Lit로 먼저, Flutter는 나중에.** → §2.1

### 2.1 UI — 웹으로 먼저 검증하고 셸은 나중에

`ui/` — **Lit 3 + Vite + TypeScript 웹컴포넌트 1,567줄**

```
wallpad-app.ts      661줄   월패드 셸
app.ts              237줄
commission-dialog.ts 150줄   BLE 커미셔닝 흐름
api.ts              114줄
device-card.ts      118줄
styles.ts           127줄   디자인 토큰
chat-panel.ts        81줄
a2ui-renderer.ts     79줄   ← 여기가 핵심
```

임베디드 타깃에 Flutter를 올리는 건 비싸다. 빌드가 길고, 반복이 느리다.
그래서 **웹컴포넌트로 먼저 만들고 브라우저에서 검증**한다.
Lit을 고른 이유는 웹컴포넌트가 **어떤 셸에도 들어가기 때문**이다 — 브라우저든, Flutter WebView든, Android든.
검증이 끝난 뒤에 Flutter가 셸을 맡는다. 순서를 바꾸면 매 반복이 몇 분씩 든다.

### 2.2 A2UI — UI를 코드가 아니라 데이터로

`docs/A2UI.md`

```
A2UI = Agent → JSON Surface → Viewer(렌더러)
```

에이전트가 **선언적 JSON으로 UI를 기술하면** 뷰어가 그린다.
여기서 "에이전트"는 AI만이 아니다.

| 경로 | 누가 UI를 결정하나 | 트리거 | 예시 |
|---|---|---|---|
| **Home Surface** | Go `surface.go` (룰 기반) | 시간·상태 변화 | "17:00", "나른한 오후예요", "기기 2개 온라인" |
| **Chat Surface** | LLM (DeepSeek / sLLM) | 사용자 질문 | "디바이스 목록" → 카드형 리스트 |

둘 다 **같은 JSON 포맷**을 뱉고 **같은 렌더러**가 그린다.

> **UI가 코드가 아닌 데이터로 선언된다. 코드를 보내면 보안 위험, 데이터를 보내면 안전하다.**

에이전트에게 화면을 그리게 하고 싶지만 코드 실행 권한은 주기 싫다.
그 긴장을 푸는 방법이 이것이다. 나는 이게 에이전트 시대 프론트엔드의 답이라고 본다.

### 2.3 곁가지 — 고객 납품과 국책과제

- **Matter 월패드 (고객 참여)** — RK3576 + Android 15, ESP32-H2를 Thread RCP로,
  AOSP 네이티브 CHIP C++ SDK + `ot-daemon`. **버전 관리되는 Android SDK(AAR)**로 고객사 네임스페이스에 납품
- **국책 R&D (IITP, 2025 ~ 2028)** — AI SoC 벤더와 함께하는 지능형 홈 과제.
  우리 범위: sLLM 음성 제어, Matter 통합, 가속기 계열 간 NPU 워크로드 포팅. 현재 2년차

### 무엇이 남았나

- **3-에이전트 병렬 개발 세션** — 하루 24 커밋, 테스트 163개 통과, **파일 충돌 0**. 사람은 PM만.
  충돌이 0인 건 모델이 똑똑해서가 아니라 **작업 경계를 미리 갈라 뒀기 때문**이다
- 리눅스/안드로이드 코드 공유 96%

## 3. AX 전환 — AI 인프라와 도메인 오너 에이전트

**고퀄 · 2025.06 ~ 현재**

### 3.1 1단계 — 인프라를 깐다 (2025.06 ~ 2025.09)

입사하고 처음 맡은 일. 사내에 AI가 돌 땅이 없었다.

- **GPU 클러스터** — RTX 5080 × 3, NixOS 25.11, 10G 네트워크, Docker 서비스 17개
- **계층형 에이전트 시스템** — n8n 워크플로 40+ 노드 + Supabase **pgvector** (문서 임베딩 2,945건)
- **데이터 파이프라인** — Airbyte 커넥터: Channel.io · Notion · JIRA → PostgreSQL **JSONB**
- **사내 포털** — Cloudflare Zero Trust 뒤

**RAG를 데모로 끝내지 않았다.** 이미 돌고 있던 낡은 사내 시스템에 붙여서,
실제 업무 데이터를 쓰는 데까지 밀었다.

### 3.2 2단계 — 에이전트를 사람에게 붙인다 (2026.05 ~ 현재)

인프라가 깔리고 나니 요청의 성격이 바뀌었다.
**"대시보드 만들어 주세요"가 아니라 "나 없이도 돌아가게 해 주세요"**였다.

그래서 대시보드를 만들지 않는다. **도메인의 주인에게 에이전트를 붙인다.**

#### VOC 워크벤치 — 정량적 가추(abduction)

매일의 고객 상담 데이터 위에서 도는 추론 도구. 운영팀이 매일 아침 읽는다.

설계에서 가장 오래 붙든 건 모델이 아니라 **숫자의 정직성**이었다.

1. **모든 숫자의 단위·기간·포함정책을 먼저 고정한다.** 이게 안 되면 나머지는 전부 헛것이다
2. **근거를 개별 대화 id까지 역추적 가능하게** 만든다. "왜 이 수치냐"에 답할 수 없으면 운영팀이 안 믿는다
3. 기준선과 **다른** 신호를 잡는다
4. 설명되지 않고 남는 **잔차를 다음 사이클의 가설**로 넘긴다

공개 실험체: [abductcli](https://github.com/junghan0611/abductcli) (Clojure) —
"놀라운 숫자에서 그것을 설명해야만 하는 숨은 규모로 거꾸로 추론하기".
사적 실험으로 시작해서, 다른 몸으로 회사의 운영 워크벤치가 됐다.

#### 인시던트 워크벤치 — 읽기 전용 증거 도구

장애 조사에서 덤프를 쌓는 순간 그 덤프가 또 하나의 거짓말이 된다.
그래서 **저장하지 않고 살아 있는 원본에서 읽는다.**
모든 시각을 **KST 단일 시간축**으로 정렬한다.
VOC → 디바이스 → 런타임으로 이어지는 사슬을 잇는다.

#### 비서실장(COS) 에이전트

내 회사 포트폴리오 자체를 얹은 작업면. 트랙, 오너, 결정, 원장(ledger).

#### Forge 커넥터

→ [forge-config](https://github.com/junghan0611/forge-config)

셀프호스팅 Forgejo 위에서 에이전트가 대화를 **리뷰 가능한 작업 항목**으로 굳힌다.
이슈 → 코멘트 → 라벨 → PR을 돌면서.

> **말 걸 때만 답하는 에이전트는 채팅창이고,
> 작업 항목을 소유하고 거기로 되돌아오는 에이전트는 동료다.**

이게 이 축 전체의 문장이다. 그리고 내가 다음으로 쌓을 축이다.

### 3.3 곁가지 — PSIRT

외부에서 신고된 취약점의 접수 · 재현 · 영향 평가 · 벤더 에스컬레이션 · 규제기관/고객 대응.
제품 보안 대응 체계를 맡고 있다.

## 4. entwurf — 에이전트 하네스 디스패치 기층

**독립 · 2025.08 ~ 현재 · npm 배포**
→ [entwurf](https://github.com/junghan0611/entwurf) · [`@junghanacs/entwurf`](https://www.npmjs.com/package/@junghanacs/entwurf)

### 문제

에이전트 하네스가 여럿이다 — Claude Code, Codex, Antigravity, pi.
이들이 서로를 부르게 하려는 시도는 보통 **하나가 나머지를 삼키는 방식**으로 끝난다.
OAuth 프록시를 세우고, CLI 대화록을 스크래핑하고, 백엔드 신원을 갈아 끼운다.
그러면 하네스 하나가 죽을 때 전부 죽는다.

### 무엇을 골랐나

**서로의 대화록·인증·런타임을 소유하지 않는다.**
정체성(garden id)만으로 서로를 호출한다. 기층은 **주소만 나른다.**

| 하네스 | 어떻게 들어오나 |
|---|---|
| Claude Code | 메타브리지 |
| pi | 컨트롤 소켓 어댑터 |
| Codex · Antigravity | 전달 검증 프로브 |

각자 자기 정체성을 유지한다.

### 왜 그것이었나

독일어 *Entwurf* = 기투(企投), 앞으로 던짐.
**"멀티 에이전트"가 하고 싶었던 게 아니다.**
위임과 연속성과 공유 도구가 **어느 한 세션보다 오래 살아남는 작업 환경**에서
어떻게 행동해야 하는가 — 그 규약을 코드로 박고 싶었다.

이름이 말해 준다. **entwurf는 일회용 일꾼이 아니라 형제를 연다.**
전달·깨움·재개·메타세션 인계를 가로질러 정체성이 보존되는 에이전트 작업 분신.

### 무엇이 남았나 — 제3자의 행동

*2026-07-10 기준*

- npm **30일 설치 1,395회**, GitHub **stars 21**
- **한 번도 만난 적 없는 개발자가 Snowflake Cortex Code ACP 백엔드를 들고 왔다.**
  [PR #40](https://github.com/junghan0611/entwurf/pull/40) — 11개 파일, +885줄.
  **내가 한 번도 써 본 적 없는 엔터프라이즈 에이전트 런타임**이다.

마지막 줄이 이 프로젝트에서 유일하게 자랑할 만한 것이다.
그는 **확장 경계를 아키텍처가 있다고 말한 바로 그 자리에서 찾아 썼다.**
추상화에 대한 리뷰 중에 값이 나가는 건 이것 하나뿐이다.

### 곁: 상류에 머지된 코드

내가 소유하지 않은 코드베이스에 받아들여진 패치 두 건.

### 5.2 문서 계약 — 글솜씨가 아니라 구조

| 파일 | 역할 |
|---|---|
| `AGENTS.md` | 리포의 영속 기준선 — 정체성, 불변식, 작업 규칙. 에이전트가 세션마다 다시 읽는다 |
| `VOCABULARY.md` | 용어 단일 진실원. 하중을 받는 낱말은 **한 번만** 정의한다 |
| `NEXT.md` | 핸드오프 — 다음 한 걸음. 세션이 끊겨도 다음 에이전트가 이어받는다 |
| `botlog` | 에이전트가 자기 작업을 스스로 org 노트로 남긴다 |

사람에게 읽히려고 쓴 문서가 아니라 **사람과 에이전트가 같이 읽는 계약서**다.

예컨대 이 저장소의 [`AGENTS.md`](../AGENTS.md)에 박힌 규칙들:

- 숫자는 100 단위로 **내림**한다. 정확한 값이 아니라 **공개 검증 가능한 근사값**이 표준이다
- 정체성 문서에 한국어가 새어 들면 **회귀(regression)**다. 스테이징 전에 매번 검사한다
- 신뢰의 근거가 될 항목은 **최소 하나는 제3자의 행동**이어야 한다
- 커밋 수를 `~/repos/` 전체 순회로 세지 마라 — 같은 리포의 클론이 중복 계수된다
  (실제로 이 함정에 빠져 90일 수치가 3,200 → 5,470으로 부풀었던 적이 있다. 그 사고를 문서에 박아 뒀다)

**AGENTS.md에 실패 사례를 적는다.** 그게 다음 에이전트를 구한다.

### 5.3 3층 크로스링궐 검색 — 한국어 RAG가 실패하는 자리

```
질의: "보편 학문에 대한 문서"

1층  임베딩 벡터      → [paideia, universalism] 태그가 붙은 노트
2층  Denote dblock 그래프 → 메타노트 정규식 → 연결된 22개 노트
3층  개인 어휘 온톨로지  → dictcli expand("보편") → [universal, paideia, liberal arts]
```

1층만으로는 **이미 영어로 태그된 노트**밖에 못 찾는다.
같은 생각을 한국어 이름으로 주장하는 노트는 끝내 보이지 않는다.
세 층을 합쳐야 일반 RAG 스택이 뭉개는 노트 생태가 복원된다.
WordNet에 없는 개인 온톨로지까지 포함해서.

→ [andenken](https://github.com/junghan0611/andenken) (LanceDB) · [dictcli](https://github.com/junghan0611/dictcli) (어휘 그래프 3,900+ 트리플, 한↔영 매핑 2,400+)

### 5.4 memex-kb — 한국 조직이 전부 부딪히는 벽

→ [memex-kb](https://github.com/junghan0611/memex-kb)

위의 모든 것은 "문서가 평문 텍스트가 될 수 있다"를 전제한다. **한국에서는 대개 안 된다.**

```
레거시 콘텐츠 → 구조화된 텍스트 → 재현 가능한 산출물 → 사람 + AI 협업
```

- `hwpx2org` — 어떤 툴체인도 건드리기 싫어하는 한글 워드프로세서 포맷
- `scanpdf2org` — 스캔된 종이를 비전 전사로
- `epub2org` · `html2epub` · `org2odtdoc` — 오피스 포맷으로의 왕복
- `textlint-ko` — 한국어 산문 린팅
- 제안서 파이프라인 — 실제로 예산을 결정하는 문서들

Org-mode를 **메타 문서**로 두고 전부 그리로 통과시킨다.
HWP, 세로형 관공서 양식, 스캔된 정부 PDF, 토크나이저 가정을 무너뜨리는 형태소.
부수적인 배관이 아니다. **한국의 모든 조직이 이 벽에 부딪힌다.**

### 5.6 공유 시간축 — 하네스가 시간을 만나는 곳

사람과 AI 에이전트가 **같은 org-agenda 뷰**를 본다. 오케스트레이션이 아니라 공유 대장간이다.

```
05:53  Human      기상
08:42  Agent(T)   doomemacs-config: feat: agent-shell 0.48.1 업그레이드
09:40  Agent(T)   agent-config: notify.ts 제거 — Emacs RPC 버그 해결!
09:52  Human      많은 것을 금새 해결
10:33  Agent(O)   geworfen: Human/Agent/Diary 통합 + org 링크 클릭
```

네 개의 소스가 하나의 시간축에서 합쳐진다 —
**Human**(저널), **Agent(T)**(로컬), **Agent(O)**(클라우드 봇), **Diary**(반복 일정).
에이전트가 커밋을 찍으면 타임라인에 뜬다. 사람이 "밥먹고 올게"라고 쓰면 에이전트는 계속 일한다.

## 6. 기반 — 이게 없으면 위의 전부가 무너진다

### 6.1 nixos-config — 어디서나 같은 기계

→ [nixos-config](https://github.com/junghan0611/nixos-config)

선언적 NixOS **4대** — 노트북(ThinkPad), NUC, Oracle ARM, RPi5.
flake 하나, `nixos-rebuild switch`, 동일한 환경.
**17개 이상의 Docker 서비스**를 전부 Nix로 선언.
기계가 죽으면 새 기계가 저장소 하나로 같은 세계를 부팅한다.

**재현성은 편의가 아니라 에이전트 신뢰의 전제조건이다.**
환경이 결정론적임을 아는 에이전트만 확신을 갖고 행동할 수 있다.

### 6.2 doomemacs-config — 울타리 철학

→ [doomemacs-config](https://github.com/junghan0611/doomemacs-config)

에디터 설정이 아니다. `agent-server.el` — 에이전트가 org-agenda를 읽고, Denote 노트를 검색하고,
서지를 질의하고, dblock을 갱신하는 **Elisp 인터페이스 10개 API**를 emacsclient 소켓으로 연다.

**에이전트를 프롬프트로 제약하지 않는다.** 대신 울타리 친 놀이터를 준다.

```
울타리 (agent-server.el)      놀이터 (에이전트의 자유)      보호자 (호스트/사람)
──────────────────────────    ───────────────────────────   ──────────────────────
경로 가드: 읽기 4개 디렉토리   REPL에서 새 함수 정의         모니터링, 복구
API: 아젠다·검색·서지·dblock   org 파싱, dblock 갱신         에스컬레이션, 재설계
쓰기: botlog + 추적 데이터만   질의 연쇄, 상호참조           최종 책임
```

울타리 안에서 에이전트는 자유롭다.
**에이전트가 뭔가 부수면 그건 에이전트 문제가 아니라 시스템 설계 문제다.**
신뢰는 감시가 아니라 구조에서 온다.

Emacs가 사람에게 보여주는 바로 그 `agent-org-agenda-day` 함수를,
Oracle Cloud의 Docker 봇도 호출하고, geworfen이 웹에도 서빙한다.
**하나의 인터페이스, 세 개의 소비자.**

## 8. 이 포트폴리오를 검증하는 법

내 저장소는 **능력**을 증명하지만 **수용**을 증명하지 못한다.
저자가 유일한 심판인 닫힌 고리일 수 있다는 의심은 정당하다.

그래서 **근거가 제3자의 행동인 항목**을 항상 유지한다.

| 항목 | 제3자 |
|---|---|
| [entwurf#40](https://github.com/junghan0611/entwurf/pull/40) | 모르는 외부 개발자가 엔터프라이즈 ACP 백엔드를 기여 (11파일, +885) |
| [dakra/ghostel#343](https://github.com/dakra/ghostel/pull/343) · [#510](https://github.com/dakra/ghostel/pull/510) | 남의 코드베이스가 내 CJK 입력 패치를 머지 |
| npm 설치 1,395회 / 30일 | 이름 모를 사용자들 |

이 줄들이 위조하기 가장 어렵고, 회의적인 독자가 가장 먼저 확인하는 것들이다.
낡으면 교체한다. 항목 자체를 조용히 지우지는 않는다.

**그리고 이 문서를 그대로 믿지 마시라.**
